# AAF 能力系统接口文档

状态：v0.4 自动化配置与伤害应用基线  
日期：2026-04-30  
适用范围：能力库、角色能力绑定、场景内能力执行、半自动/全自动结算

> 本文档取代旧版“能力直接触发公式并立即结算”的描述。后续实现一律以“AbilityDefinition + AbilityWorkflowRun + 自动化模式”作为主线；旧文档中没有 workflow 的执行流程视为过时。

## 1. 核心原则

能力是规则意义上的统一实体。种族、职业、职业特性、背景、物品、武器、法术、战技、状态来源都可以表现为能力；其中一部分只提供描述或绑定关系，一部分参与自动结算。

AAF 的能力系统分三层：

1. `AbilityDefinition`：GM 在能力库中创建的静态定义，包含名称、描述、来源、条件、资源、检定、伤害、效果和默认自动化配置。
2. `CharacterAbilityInstance`：角色实际拥有的能力实例，后续用于记录来源、等级、启用状态、次数、覆盖值和绑定关系。
3. `AbilityWorkflowRun`：一次能力执行的运行记录，描述声明、选目标、扣资源、检定、伤害、应用效果、日志、撤销快照等阶段。

## 2. 自动化模式

`AbilityAutomationMode` 目前有三档：

| 模式 | 用途 | 后端行为 |
| --- | --- | --- |
| `manual` 手动预览 | 类似 FVTT/dnd5e 默认体验，先生成结算卡和流程记录 | 计算公式和掷骰预览，但不真实扣资源、不扣血、不应用效果，workflow 标记等待手动确认 |
| `assisted` 半自动 | 默认内部测试模式，GM/玩家可以看见流程并保留修正空间 | 自动掷骰、扣资源、应用伤害和效果，同时保留 workflow 与撤销快照 |
| `full` 全自动 | 类似 midi-qol 的一条龙结算 | 自动确认目标、掷骰、判定命中/豁免、应用伤害和效果，默认不等待人工确认 |

世界级开关以后应映射到这三档，而不是额外发明一套“开/关”布尔值：关闭自动化使用 `manual` 或低自动化 `assisted` 配置，开启自动化使用 `full` 或高自动化 `assisted` 配置。

## 3. Workflow 阶段

`AbilityWorkflowRun.phases` 固定使用下列阶段。没有用到的阶段必须标记为 `skipped`，不能直接省略。

| 阶段 | 说明 |
| --- | --- |
| `declare` | 声明使用能力，记录施放者、能力和模式 |
| `target-confirmation` | 选择并确认目标 |
| `cost-check` | 检查并消耗 MP、怒气、物品、次数等资源 |
| `reaction-window` | 反应、打断、借机效果窗口，当前先保留接口 |
| `attack-roll` | 攻击检定 |
| `save-roll` | 豁免或对抗检定 |
| `damage-roll` | 伤害骰和伤害类型 |
| `damage-application` | 抗性、免疫、易伤、减伤、临时 HP、实际扣血/预览 |
| `effect-application` | 状态、标签、属性修正、治疗、临时 HP 等效果 |
| `post-apply` | 浓度、持续伤害、回合后钩子，当前先保留接口 |
| `settle` | 完成日志、快照和最终状态 |

每个阶段有 `automation: manual | prompt | auto` 和 `status: pending | waiting | skipped | resolved | failed`。前端应根据这两个字段决定是否显示确认、编辑和应用按钮。

## 5. 执行接口

场景内执行能力：

`POST /api/worlds/:worldId/scenes/:sceneId/abilities/:abilityId/execute`

请求体：

```json
{
  "actorCharacterId": "character-id",
  "targetCharacterIds": ["target-id"],
  "automationMode": "assisted",
  "automation": {
    "autoApplyDamage": true,
    "allowManualOverride": true
  },
  "manualOverrides": {
    "attackRoll": 17,
    "damage": 8
  },
  "metadata": {
    "source": "world-battle-panel"
  }
}
```

返回体的 `data` 必须包含：

- `ability`
- `actor`
- `targets`
- `settlement`
- `costs`
- `effects`
- `workflow`
- `logEntry`

`workflow` 同时写入场景 `canvasState.abilityLogs`，用于后续聊天卡片、撤销、审计和重放。

`workflow.damageApplications[]` 记录每个目标的 `rawDamage`、`effectiveDamage`、`tempHpDamage`、`hpDamage`、`oldHp/newHp`、`oldTempHp/newTempHp`、`resistanceApplied`、`vulnerabilityApplied`、`immunityApplied`、`flatReduction` 和 `damageTypeModifiers`。`manual` 或关闭自动应用伤害时，记录仍保留预览数值，但 `applied` 为 `false` 且不会改写角色 HP。

## 6. 能力字段约定

`AbilityDefinition` 当前仍以单活动能力为主，后续可以升级为 `activities[]`，但执行接口和 workflow 不应被推翻。

关键字段：

- 基础：`name`、`category`、`source`、`sourceName`、`activation`、`actionType`、`description`、`rulesText`、`tags`
- 绑定：`levelRequirement`、`professionLevelRequirement`、未来 `bindingRules`
- 范围：`range`、`aoeShape`、`aoeSize`
- 资源：`resourceCosts`、`cooldown`
- 持续：`duration`、`durationValue`、`concentration`
- 检定：`checkType`、`attackAttribute`/`attackAttr`、`saveDC`
- 伤害：`damageRolls`
- 触发：`trigger.timing`、`trigger.condition`、`trigger.limit`、`trigger.priority`
- 效果：`effects`
- 反应：`reactionStrategy`
- 自动化：`automation?: Partial<AbilityAutomationConfig>`

`automation` 已作为 `AbilityDefinition` 的持久化 JSON 字段进入 Prisma、导入导出和能力编辑器。执行接口收到本次请求级 `automation` / `automationMode` 时以请求为准；没有请求覆盖时使用能力定义中的默认配置，再回退到 `assisted` 预设。

GM 侧公式录入必须优先使用中文选项和注册表映射，例如“力量调整值 + 敏捷调整值 >= 固定 DC 15”。不要要求 GM 手写任意 JS 或不透明脚本。

## 7. 状态分类

状态不只是单个 key，还必须有分类。规则书中的“定身、干扰、其他状态、效果标记”等分类在代码中有意义：

- `hasStatusCategory`
- `removeStatusCategory`
- `grantStatusCategoryImmunity`

当能力写“解除定身”或“免疫定身”时，含义是命中该分类下的任意状态，而不是只匹配某一个具体状态名。

## 8. 当前限制

- 初版只把一次能力视为单活动；多活动能力、武器多段攻击、召唤、变形后续再扩展。
- `reaction-window`、`post-apply`、浓度、持续伤害和撤销按钮目前是接口预留。
- 伤害应用当前只对主目标结算；多目标 AOE 的逐目标伤害应用仍待扩展。
- `manualOverrides` 已进入接口，但当前只作为 workflow 记录，尚未完全接入每个掷骰和数值节点。

## 9. 后续实现顺序

1. 把角色拥有的能力实例、职业等级解锁和能力绑定规则落库。
2. 实装 workflow 卡片确认：手动应用伤害、手动应用效果、撤销本次执行。
3. 扩展多目标 `damage-application` 与治疗反制。
4. 增加反应窗口、浓度、回合开始/结束触发器和持续效果队列。
