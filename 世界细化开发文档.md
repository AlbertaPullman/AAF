# AAF 世界细化开发文档（阶段 8 实施主线）

本文档是世界系统的实施主线文档。

适用范围：以“一个世界 = 一个完整团务包”为核心，对世界内的功能拆分、数据模型、接口、事件、实现顺序与验收标准进行工程化落地。

主参考来源：世界需求与填写清单.md。

---

## 1. 开发总目标

### 1.1 产品目标

1. 世界不是单一战斗地图，而是完整团务系统。
2. 玩家与 GM 的主要操作都在世界内完成，不依赖外部工具界面。
3. 世界要支持长期运行、动态扩展、强规则驱动。

### 1.2 研发目标

1. 采用模块化与组件化实现，保证功能可扩展。
2. 以中文化体验为默认标准，所有核心交互都提供中文文案。
3. 先落地 P0 闭环，再逐步扩展 P1/P2。

### 1.3 本阶段核心冻结项（来自需求清单）

1. 战斗场景核心闭环。
2. 智能角色卡与结算闭环。
3. 天赋树核心闭环。
4. 可视化规则编辑器 MVP（含光源与测量板模板）。
5. 数据包基础能力。

## 2. 世界架构分层

### 2.1 世界内分层

1. 展示层：场景视图、面板、卡片、编辑器。
2. 应用层：状态编排、权限判断、流程控制。
3. 规则层：结算引擎、触发器引擎、天赋树引擎。
4. 数据层：角色、物品、能力、任务、日历、战斗记录。
5. 集成层：酒馆适配、音视频资源、扩展包管理。

### 2.2 模块边界

1. 战斗模块：地图、视野、先攻、回合、战斗卡片。
2. 角色模块：角色卡、装备槽、能力槽、资源状态。
3. 天赋模块：节点解锁、点数消耗、效果注入。
4. 规则编辑模块：条件动作编排、模板系统、沙盒预览。
5. 任务日历模块：任务树、时间推进、提醒。
6. 数据包模块：基础包、扩展包、版本与依赖。
7. AI 模块：聊天桥接、总结草案、审核落库。

### 2.3 模块通信规范

1. 默认通过公开服务接口通信。
2. 跨模块事件通过事件总线广播。
3. 禁止在业务代码中直接读写其他模块私有状态。
4. 调试接口只在开发模式可用。

## 3. 首发 P0 子集实施范围

### 3.1 P0-1 战斗场景核心闭环

目标：完成可实战的地图与回合系统。

范围：

1. 网格地图、背景图、基础绘制工具。
2. 动态光源与迷雾。
3. 视野计算（角色感知能力 + GM 覆盖）。
4. 先攻列表与回合推进。
5. 战斗信息频道卡片化输出（简版 + 详情）。

### 3.2 P0-2 智能角色卡与结算闭环

目标：角色卡不只是展示，而是可执行结算。

范围：

1. 属性、资源、装备、能力、冷却、等级、经验。
2. 装备拖拽到槽位后自动更新动作面板。
3. 攻击/伤害自动结算与多段伤害展示。
4. 反应触发策略（询问 / 自动）可配置。
5. 怪物关键数值隐藏策略（由 GM 控制）。

### 3.3 P0-3 天赋树核心闭环

目标：天赋树直接驱动能力解锁与结算。

范围：

1. 多类型天赋点（职业/通用/种族/世界自定义）。
2. 节点解锁规则（复杂条件表达式）。
3. 多树并行学习与洗点。
4. 天赋效果注入结算引擎。

当前实现状态（2026-03-23）：

1. 大厅侧已具备模板级编辑器（X6）与只读预览投射能力。
2. 模板支持职业/通用树类型、自定义分类、发布版本与模板删除。
3. 预览已支持真实节点坐标与连线渲染，节点详情位于右侧详情栏。
4. 世界内尚未接入“实例化树 + 玩家分配 + 结算联动”。

世界内联动实施顺序（必须按序落地）：

1. 世界只读接入：在世界页复用大厅预览投射组件，先实现稳定只读查看。
2. 实例化层：新增 WorldTalentTreeInstance（模板快照 + 版本锁定 + 启用状态）。
3. 分配层：新增 CharacterTalentAllocation（节点解锁、点数消耗、洗点记录）。
4. 投影层：将已解锁天赋效果写入 CharacterRuntime（属性、能力开关、触发器参数）。
5. 结算层：战斗/规则引擎统一读取 CharacterRuntime 天赋投影，参与命中、伤害、资源等计算。
6. 治理层：支持重算与回滚（洗点、模板升级迁移、版本冲突处理）并记录审计日志。

数据一致性约束：

1. 世界对模板必须走快照，不允许直接引用可变模板内容。
2. 天赋效果计算要幂等，角色重连后可完整重放。
3. 模板版本升级仅影响新实例，旧实例通过迁移脚本显式升级。

### 3.4 P0-4 可视化规则编辑器 MVP

目标：GM 可在世界内以中文界面编排规则。

范围：

1. 条件动作式可视化编辑。
2. 全中文节点、参数、错误提示。
3. 光源模板（含光亮术示例）。
4. 测量板模板（命名、颜色、常驻、清理）。
5. 沙盒预览与一键测试事件。

### 3.5 P0-5 数据包基础能力

目标：保证规则资产可管理、可扩展。

范围：

1. 基础包强制启用。
2. 扩展包导入、启停。
3. 冲突处理优先基础包。
4. 基础版本检查与依赖校验。

## 4. 数据模型落地建议

### 4.1 世界核心实体

1. WorldRuntimeState：世界运行状态（active/sleeping）。
2. SceneState：场景状态快照（对象、光源、迷雾、触发器）。
3. CombatState：先攻、轮次、回合、暂停原因。
4. CharacterRuntime：角色实时状态（资源、状态效果、触发配置）。
5. TalentAllocation：天赋点分配与节点解锁记录。
6. RuleGraph：可视化规则图定义（条件节点、动作节点、参数）。
7. RuleTemplate：规则模板仓库（光亮术、测量板等）。
8. PackageManifest：基础包与扩展包清单、版本、依赖。

### 4.2 AI 相关实体

1. WorldLoreEntry：世界书条目。
2. SceneLoreEntry：场景级世界书条目。
3. SummaryDraft：总结草案。
4. SummaryDraftItem：草案明细项（候选变更）。
5. SummaryReviewLog：审核日志。

### 4.3 中文化实体支持

1. DisplayNameZh：中文显示名。
2. DescriptionZh：中文描述。
3. UiLabelZh：中文界面标签。
4. AliasEn（可选）：调试映射英文别名，仅 GM 调试可见。

## 5. 接口与事件落地建议

### 5.1 世界内 API 分组

1. 场景组：地图、光源、迷雾、测量板、对象。
2. 战斗组：先攻、回合推进、战斗卡片、隐藏规则。
3. 角色组：角色卡、装备槽、能力配置、反应策略。
4. 天赋组：树定义、节点解锁、洗点。
5. 规则组：规则图、模板、沙盒执行。
6. 数据包组：导入、启停、依赖校验。
7. AI 组：聊天桥接、总结生成、草案审核。

### 5.2 关键 Socket 事件

1. 战斗事件：combat:initiative:roll、combat:turn:next、combat:card:new。
2. 场景事件：scene:light:update、scene:fog:update、scene:measure:add。
3. 规则事件：rule:test:run、rule:test:result。
4. AI 事件：world:ai-message:new、world:summary:ready。

### 5.3 事件设计约束

1. 事件名采用 domain:entity:action。
2. payload 必须带 worldId 与 sceneId（适用时）。
3. 所有失败响应统一 ack error 结构。

## 6. 开发阶段拆分

### 6.1 迭代 A（基础框架）

1. 世界运行时状态管理。
2. 模块注册机制与事件总线。
3. 中文文案基础框架。

验收：

1. 模块可独立注册、卸载。
2. 世界进入后可按需懒加载模块。

### 6.2 迭代 B（战斗闭环）

当前执行状态（2026-03-13）：已完成后端首段开工与前端首轮接入，落地场景视觉状态（网格/光源/迷雾）与战斗状态（先攻/轮次/回合）API，并在世界页提供 visual/combat 面板最小交互（刷新、更新、next-turn）。下一步进入战斗卡片输出与更细粒度交互编排。

1. 地图、网格、视野、迷雾、光源。
2. 先攻列表与回合推进。
3. 战斗卡片输出。

验收：

1. 至少支持 10 人、30 token 的稳定回合结算。
2. 可隐藏怪物关键数值。

### 6.3 迭代 C（角色与天赋）

1. 角色卡执行化（装备联动动作）。
2. 天赋树解锁与洗点。
3. 天赋效果接入结算。

验收：

1. 装备变化可实时反映到动作与结算。
2. 天赋解锁可直接影响战斗结果。

### 6.4 迭代 D（可视化编辑器）

1. 条件动作编辑器。
2. 光亮术模板。
3. 测量板模板。
4. 沙盒测试。

验收：

1. GM 可独立创建并调试规则。
2. 光亮术与测量板模板可在实战场景复用。

### 6.5 迭代 E（数据包 + AI）

1. 扩展包导入与依赖校验。
2. AI 聊天桥接。
3. 总结草案与审核流。

验收：

1. 扩展包冲突时按基础包优先。
2. 总结不会自动写入正式设定。

## 7. 非功能要求

### 7.1 性能目标

1. 单场景并发：10 人。
2. 单场景 token：30。
3. 关键交互延迟：主要操作应保持可接受实时反馈。

### 7.2 稳定性目标

1. AI 服务不可用不影响世界主流程。
2. 规则执行失败可回滚并提示 GM。
3. 音视频加载失败要有公告与降级处理。

### 7.3 可维护性目标

1. 所有新增能力必须有中文文案和说明。
2. 规则模板与业务逻辑分离。
3. 核心规则函数优先纯函数化并配套单测。

---

## 8. 风险与对策

### 8.1 主要风险

1. 规则编辑器过于宽泛导致实现失控。
2. 战斗反应链过长导致结算阻塞。
3. 模块耦合过深导致回归风险上升。
4. 中文化术语不统一导致学习成本上升。

### 8.2 对策

1. 模板先行：先实现模板化能力，再放开自由编排。
2. 阻塞控制：反应触发提供默认自动策略。
3. 接口收口：跨模块只通过服务接口与事件总线。
4. 术语词典：建立统一中文术语表并在 UI 强制使用。

## 9. 执行与文档同步规则

1. 任何世界功能开发，必须先对照本文档所属迭代。
2. 新增接口或事件必须同步更新 docs/interface-registry.json。
3. 新增规则名词必须同步更新 docs/naming-convention.md 或世界术语清单。
4. 每次完成一个子任务，必须更新 AI上下文记忆文档.md 与 docs/change-log.md。
