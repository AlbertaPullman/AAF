# 阶段 8 执行任务清单 - AI 酒馆桥接、NPC 记忆与主持人审核流

当前补充状态（世界模块）：

3. 角色卡、世界书、会话记录、审核结果一律以 AAF 数据为准。
4. 总结器只允许主持人手动触发，不允许自动覆写设定。

## 2. 架构目标

### 2.1 阶段 8 的核心定位

阶段 8 不是“把酒馆嵌入网站”，而是“让 AAF 通过适配层调用酒馆能力”。

推荐调用链：

1. 玩家或主持人在 AAF 世界内聊天中发言。
2. AAF 后端根据当前世界、场景、NPC、世界书整理上下文。
3. AAF 通过 tavern adapter 调用 SillyTavern。
4. SillyTavern 再调用实际模型 API。
5. 返回的 AI 回复由 AAF 写回自己的世界消息流。

### 2.2 职责边界

#### AAF 负责

1. 用户、世界、场景、角色、消息、权限。
2. NPC 角色卡编辑与世界书编辑。
3. AI 调用前的上下文整理与调用审计。
4. AI 回复回写、草案存储、主持人审核。
5. 结构化总结结果的确认与正式落库。

#### SillyTavern 负责

1. 接收 AAF 传来的 prompt 上下文。
2. 管理角色预设、世界书注入策略、推理预设。
3. 调用外部模型 API。
4. 返回生成结果给 AAF。

#### 模型 API 负责

1. 实际生成聊天回复、摘要、候选记忆更新。

### 2.3 第一版落地原则

1. 第一版由 SillyTavern 调用模型 API，AAF 不直接接模型。
2. AAF 与酒馆之间必须通过独立适配层交互，不能把酒馆逻辑散落在聊天 controller 或 socket 里。
3. 酒馆不可成为唯一记忆源；任何关键设定必须在 AAF 可追溯。

## 3. 功能范围

### 3.1 世界内 AI 聊天桥接

目标：让世界内聊天可以对接指定 NPC，由 AAF 把消息转发给酒馆并回收回复。

最小行为：

1. 玩家在世界聊天中选择要对话的 NPC。
2. 当前消息、最近聊天记录、当前场景上下文、NPC 设定被整理后发送到 tavern adapter。
3. AI 回复作为一条新的世界消息进入当前 scene 的聊天流。
4. AI 回复消息需要带上明确来源标记，例如 `fromCharacterId` 或 `messageType=AI_NPC`。

### 3.2 NPC 角色卡同步

目标：主持人在 AAF 编辑 NPC 时，AAF 自动把可用于扮演的数据同步给酒馆。

建议同步的数据：

1. 基础身份：名称、称呼、头像、阵营、职业。
2. 人格设定：性格、口癖、偏好、厌恶、禁忌。
3. 行为目标：短期目标、长期目标、秘密、立场。
4. 关系设定：对玩家角色或关键 NPC 的态度。
5. 扮演约束：不会说什么、不能知道什么、对哪些设定保持未知。

### 3.3 世界书同步

目标：把 AAF 中的设定分层下发给酒馆，而不是让主持人去酒馆里手工维护。

推荐分层：

1. 全局世界书：平台级、系统级、规则级设定。
2. 世界级世界书：当前团共享的背景设定。
3. 场景级世界书：当前 scene 的临时信息、地点特征、事件状态。
4. 角色级补充：仅对某个 NPC 生效的附加设定。

### 3.4 主持人手动总结器

目标：每次聚会结束后，主持人可手动触发总结，并把结果作为“待审核草案”管理。

流程要求：

1. 只有主持人可触发总结。
2. 总结范围可按世界、场景、时间范围、消息范围选择。
3. 总结输出不得直接覆盖正式设定。
4. 总结结果进入待审核区，逐条确认后才能写入世界书或 NPC 记忆。

## 4. 推荐子任务顺序

### 4.1 子任务 1：酒馆适配层基础骨架

1. 扩展 tavern 模块，不再只返回 status。
2. 新增 tavern adapter service，统一封装 AAF 与 SillyTavern 的交互。
3. 支持健康检查、连通性检查、基础鉴权配置。

### 4.2 子任务 2：世界内聊天桥接 MVP

1. 增加“对 NPC 发话”的请求链路。
2. AAF 组装最小上下文并请求 tavern adapter。
3. AI 回复回写到世界聊天历史和 socket 广播中。

### 4.3 子任务 3：NPC 角色卡同步

1. 在角色卡中区分普通角色与 AI NPC。
2. 新增 AI 配置字段与同步逻辑。
3. 主持人修改 NPC 设定后，AAF 自动更新 tavern 侧角色模板。

### 4.4 子任务 4：世界书同步

1. 新增世界书实体或配置结构。
2. 支持世界级和场景级世界书增删改查。
3. 下发到 tavern adapter 参与上下文注入。

### 4.5 子任务 5：主持人总结器

1. 新增手动触发总结接口。
2. 把聊天记录和上下文发给 tavern adapter 生成结构化摘要。
3. 保存为待审核草案。

### 4.6 子任务 6：审核与落库

1. 主持人查看候选摘要、候选世界书变更、候选 NPC 记忆变更。
2. 支持接受、拒绝、局部编辑。
3. 确认后才写入正式世界书或 NPC 记忆。

## 5. 数据设计建议

### 5.1 建议新增的概念对象

1. TavernConnectionConfig：AAF 到酒馆的连接配置。
2. Character.aiConfig：NPC 的 AI 扮演配置。
3. WorldLoreEntry：世界书条目。
4. SceneLoreEntry：场景级世界书条目。
5. AiConversationSession：AI 对话会话映射。
6. SummaryDraft：主持人触发后生成的待审核草案。
7. MemoryPatchDraft：候选记忆更新项。

### 5.2 不建议第一版新增的复杂项

1. 自动学习型长期记忆自治系统。
2. 无审核的世界书自动写回。
3. 多模型路由和成本自动优化。
4. 完整的多代理协作系统。

## 6. 接口与事件设计建议

### 6.1 酒馆模块接口建议

建议新增最小接口：

1. `GET /api/tavern/status`
   - 查看酒馆连接状态、版本、启用状态。
2. `POST /api/tavern/test`
   - 测试 AAF 到酒馆的桥接调用是否可用。
3. `POST /api/tavern/chat`
   - AAF 把整理后的上下文发送给 tavern adapter，请求 AI 生成回复。
4. `POST /api/tavern/characters/sync`
   - 把某个 NPC 的 AI 设定同步给酒馆。
5. `POST /api/tavern/lore/sync`
   - 把世界书或场景书同步给酒馆。
6. `POST /api/tavern/summaries/generate`
   - 手动触发总结，返回结构化草案。

### 6.2 世界内业务接口建议

建议新增最小业务接口：

1. `POST /api/worlds/:worldId/ai-chat`
   - 玩家或主持人对指定 AI NPC 发话。
2. `GET /api/worlds/:worldId/lore`
   - 查询世界级世界书。
3. `POST /api/worlds/:worldId/lore`
   - 新建世界级世界书条目。
4. `GET /api/worlds/:worldId/scenes/:sceneId/lore`
   - 查询场景级世界书。
5. `POST /api/worlds/:worldId/scenes/:sceneId/lore`
   - 新建场景级世界书条目。
6. `POST /api/worlds/:worldId/summaries`
   - 主持人手动触发总结任务。
7. `GET /api/worlds/:worldId/summaries/drafts`
   - 查看待审核总结草案。
8. `POST /api/worlds/:worldId/summaries/drafts/:draftId/apply`
   - 主持人审核后正式应用。

### 6.3 Socket 事件建议

第一版可以不急着加很多新事件，先保证 HTTP 可跑通。若要补实时反馈，可考虑：

1. `world:ai-message:request`
2. `world:ai-message:pending`
3. `world:ai-message:new`
4. `world:summary:ready`

## 7. 上下文组装策略

### 7.1 AI 聊天最小上下文

建议由 AAF 负责组装：

1. 当前世界基础信息。
2. 当前场景信息。
3. 当前 NPC 的角色设定。
4. 与当前 scene 相关的世界书条目。
5. 最近 N 条世界聊天记录。
6. 当前发话玩家的身份信息与角色信息。

### 7.2 总结器上下文

建议输入：

1. 本次会话消息范围。
2. 当前世界与场景设定。
3. 涉及的主要 NPC 列表。
4. 已存在的世界书条目摘要。

建议输出结构：

1. 会话摘要。
2. 候选世界事实更新。
3. 候选 NPC 记忆更新。
4. 候选角色关系变化。
5. 需要主持人确认的冲突项。

## 8. 审核流要求

### 8.1 主持人审核原则

1. AI 生成内容默认是候选草案，不是正式设定。
2. 每条候选变更都必须可单独接受或拒绝。
3. 允许主持人手动编辑后再保存。
4. 审核过程必须保留操作记录。

### 8.2 需要避免的风险

1. NPC 记忆被错误总结污染。
2. 场景临时信息误写成长期世界设定。
3. AI 根据过期世界书继续扮演导致串戏。
4. 玩家越权触发主持人专属总结流程。

## 9. 验收要点

### 9.1 聊天桥接验收

1. AAF 可成功调用酒馆并回收 AI 回复。
2. AI 回复能够进入当前世界和当前 scene 的聊天流。
3. 酒馆不可用时，不会拖垮世界聊天主链路。

### 9.2 设定同步验收

1. 主持人在 AAF 修改 NPC 设定后，可同步到酒馆侧。
2. 世界级和场景级世界书能参与当前 AI 对话。
3. 场景切换后，注入的世界书上下文会随 scene 变化。

### 9.3 总结审核验收

1. 主持人可手动触发总结。
2. 总结结果先进入草案区，而不是直接生效。
3. 主持人逐条审核后，正式设定才会更新。

## 10. 实施注意事项

1. 不要把 SillyTavern 当作业务数据库。
2. 不要把 tavern adapter 直接写进 chat controller 或 socket 事件处理函数里。
3. 不要在第一版就做自动记忆学习和自动世界书覆写。
4. 不要要求玩家理解或操作酒馆界面。
5. 酒馆不可用时，AAF 的账号、世界、聊天、VTT 主链路必须仍然可用。
