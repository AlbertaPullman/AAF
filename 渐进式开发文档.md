# FVTT 风格 TRPG 平台渐进式开发文档

本文档用于指导这个项目按照可跟踪、可验收、可回滚的方式逐步开发。目标不是一次性把所有功能堆出来，而是先建立稳定骨架，再逐步增加社交、世界管理、VTT、AI 能力。每个阶段都必须有明确产出、验收标准、风险点和下一步衔接方式，这样后续任何一次新的 AI 对话都可以直接接着做，而不是重新猜需求。

## 1. 项目总目标

这个项目的最终目标，是做一个面向自定义 TRPG 规则的网页版虚拟桌面平台，核心价值有三点：

1. 提供可多人实时协作的线上跑团空间。
2. 提供适合 GM 和玩家使用的基础社交与世界管理系统。
3. 把 AI 作为平台内原生能力接入，而不是外挂工具。

项目初期以 MVP 为目标。MVP 不要求把所有理想功能一次做完，但必须能够完整跑通一条主链路：

用户注册/登录 -> 进入大厅 -> 创建世界 -> 加入世界 -> 进入一个最简 VTT 房间 -> 拖动 Token 并实时同步 -> 在聊天室发消息。

只有这条链路稳定后，才进入后续功能叠加。

## 2. 开发原则

### 2.1 总体原则

1. 先做骨架，再做复杂功能。
2. 先保证数据模型和接口边界稳定，再做 UI 丰富度。
3. 先做最小闭环，再做扩展能力。
4. 实时同步优先传输增量事件，不直接频繁传整张画布 JSON。
5. AI 功能必须在主业务稳定后接入，避免过早把问题复杂化。

### 2.2 每阶段交付规则

每一个阶段完成时，都必须同步完成以下事项：

1. 更新当前阶段状态。
2. 记录新增文件、关键接口、关键数据表或字段。
3. 记录已知问题和下一阶段前置条件。
4. 更新 AI 上下文记忆文档。

## 3. 推荐总体阶段划分

建议把开发拆成 10 个阶段：

1. 阶段 0：需求冻结与工程规划。
2. 阶段 1：项目骨架初始化。
3. 阶段 2：数据库与后端基础设施。
4. 阶段 3：认证系统与前端登录流程。
5. 阶段 4：大厅与世界管理。
6. 阶段 5：全局聊天与基础 WebSocket 架构。
7. 阶段 6：最小 VTT 场景与 Token 实时同步。
8. 阶段 7：场景系统、角色卡与世界内聊天。
9. 阶段 8：AI 酒馆与 GM 助手。
10. 阶段 9：安全、部署、性能与上线准备。

下面按阶段展开详细说明。

## 4. 阶段 0：需求冻结与工程规划

这一阶段不写业务代码，目标是把后续实现边界讲清楚，避免写到一半发现需求变形。

### 4.1 要完成的事情

1. 确认项目名称、仓库结构、前后端目录命名方式。
2. 确认 MVP 范围，不把好友系统、AI 酒馆、复杂画布工具一开始全纳入。
3. 确认第一版用户角色只有 GM 和 Player。
4. 确认第一版世界权限至少支持 PUBLIC 和 PASSWORD，FRIENDS 与 PRIVATE 可先保留字段但后置实现。
5. 确认 VTT 第一版只做一个默认场景、一个可拖拽 Token、一个同步事件通道。
6. 确认 AI 能力在阶段 8 接入，不提前耦合到注册、世界、画布逻辑中。

### 4.2 本阶段输出物

1. 技术栈文档。
2. 项目概述文档。
3. 本渐进式开发文档。
4. AI 上下文记忆文档。

### 4.3 验收标准

1. 新的 AI 对话只看这四类文档，就能知道项目目标、优先级和开发顺序。
2. 已经明确第一版不能超范围开发。

### 4.4 阶段 0 冻结结果（v0.1，2026-03-11）

以下内容以用户填写的需求清单为准，并覆盖本文件早期“建议性”描述。

1. 项目名称冻结为：AAF，中文名“艾尔泽兰特：天穹之剑与命运之星”。
2. 用户规模目标：固定朋友圈，约 30 到 50 人，允许部分公开围观。
3. 平台角色冻结为：MASTER / Admin / Player（平台级）+ GM（世界级），并明确平台后台为 P0 必须项。
4. 账号体系冻结为：用户名+密码登录，用户名不可修改，支持管理员辅助找回密码，默认长登录态（180 天）。
5. 世界系统冻结为：PUBLIC / PASSWORD / FRIENDS / PRIVATE 在 P0 必须支持，不再后置。
6. 大厅与社交目标冻结为：第一版即包含世界管理、好友、私聊、公开聊天、系统公告。
7. 世界聊天室冻结为：支持世界下多子频道，默认通用频道与战斗信息频道，允许 GM 自定义增删与隐藏。
8. 聊天能力冻结为：文本、表情、图片、结算卡片、系统消息，消息历史永久保存，默认展示最近 15 天。
9. VTT 核心冻结为：背景、Token 拖拽同步、缩放平移、网格、测量尺、基础绘图工具均为 P0。
10. Token 与角色卡冻结为：Token 必须绑定角色卡，玩家可编辑自己的角色卡，角色卡快照为 P0。
11. AI 能力冻结为：AI 酒馆不进入 MVP（后置），GM 助手为 P2（后置），超预算时仅 GM 可使用 AI。
12. 性能目标冻结为：全站并发 50 到 100，单世界 10 人，Token 50ms，聊天 100ms，场景切换 200ms。
13. 部署环境冻结为：Windows 桌面优先，初始 2 核 4G，先允许公网 IP 直连。
14. 参数补充冻结为：头像上限 50MB，单世界场景上限 500，SLA 99.9%，邀请链接 365 天，JWT 365 天。
15. 架构组织冻结为：前端、后端、数据库目录必须分离；酒馆功能单独目录隔离；世界内功能按组件模块独立开发。

### 4.5 阶段 0 已闭合项（v0.1 最终）

以下项目已通过聊天补充并完成统一。

1. 头像上传上限：50MB。
2. 单世界场景上限：500。
3. SLA：99.9%。
4. 邀请链接有效期统一为天，数值 365。
5. JWT 有效期统一为天，数值 365。
6. BGM 不进入 MVP。
7. AI 酒馆不进入 MVP。
8. 最终确认通过“聊天确认”方式补齐。

## 5. 阶段 1：项目骨架初始化

这一阶段的目标是先把前后端工程跑起来，不碰复杂业务，确保后续所有模块都有落脚点。

### 5.1 目录设计

建议采用 monorepo 结构：

```text
AAF/
  client/
  server/
  data/            # 数据目录，单独存放 sqlite、备份、上传资源索引
  shared/          # 可选，用于共享类型、常量、事件名
  docs/            # 可选，后续沉淀接口文档、流程图
  技术栈.md
  项目文档.md
  渐进式开发文档.md
  AI上下文记忆文档.md
```

如果一开始不建立 shared，也要至少先保留未来可拆分的空间。

目录强约束：

1. 前端代码只放在 client，后端代码只放在 server，数据库文件与备份只放在 data。
2. 酒馆能力必须放在独立模块目录，便于后续快速替换成外部 API 调用或整体禁用。
3. 世界内能力必须组件模块化开发，例如测量板、绘图工具、Token 控制器都独立目录和独立状态。

### 5.2 前端初始化步骤

1. 创建 React 18 项目。
2. 配置 React Router。
3. 集成 Tailwind CSS。
4. 创建基础页面骨架。
5. 创建统一请求层。
6. 创建全局状态管理入口。

### 5.3 前端本阶段最低页面

1. 登录页。
2. 注册页。
3. 大厅页占位。
4. 世界页占位。
5. 404 页面。

### 5.4 后端初始化步骤

1. 创建 Node.js + Express 项目。
2. 集成 dotenv、cors、helmet、日志中间件。
3. 搭建统一错误处理中间件。
4. 搭建统一 API 返回格式。
5. 初始化 Socket.IO 服务，但这一阶段先只保证服务能启动。
6. 预留 Prisma 目录和环境变量文件。

### 5.5 通用规范要先建立

1. 环境变量命名规范。
2. 接口前缀规范，例如 /api/auth、/api/worlds。
3. 错误码或错误消息结构。
4. Socket 事件命名规则，例如 global:message、world:join、scene:token-move。
5. 上传目录规范。

### 5.6 本阶段输出物

1. 可启动的前端工程。
2. 可启动的后端工程。
3. README 或运行说明。
4. .env.example。

### 5.7 验收标准

1. 前后端都能独立启动。
2. 前端能访问后端健康检查接口。
3. 工作区结构稳定，不需要后续大改目录。

### 5.8 风险点

1. 如果一开始目录混乱，后续多人模块会很难接。
2. 如果返回格式不统一，后续前端错误处理会变脆弱。

## 6. 阶段 2：数据库与后端基础设施

这一阶段的目标是先把数据库模型和后端分层定住。这里如果做乱，后面会一直返工。

### 6.1 先做数据库建模，不先做页面

优先建立 Prisma schema，先把核心表确定：

1. User。
2. World。
3. WorldMember。
4. Scene。
5. Character。
6. Message。
7. Friend。
8. AiSession。

### 6.2 MVP 阶段真正要用到的表

第一批真正会参与运行的是：

1. User。
2. World。
3. WorldMember。
4. Scene。
5. Message。

Friend、Character、AiSession 可以先建表，后续再全面接入逻辑。

### 6.3 后端分层建议

后端至少拆成以下层：

1. routes：只负责路由绑定。
2. controllers：负责请求参数提取与响应。
3. services：负责业务逻辑。
4. repositories 或直接 Prisma 调用层：负责数据库交互。
5. middlewares：鉴权、限流、错误处理。
6. sockets：实时事件处理。

### 6.4 这一阶段要落实的能力

1. Prisma 初始化与 SQLite 连接。
2. 首个迁移文件生成。
3. 种子数据机制，可选但推荐。
4. 健康检查接口。
5. 基础日志落地。
6. 统一鉴权中间件雏形。

### 6.5 本阶段输出物

1. schema.prisma。
2. 首批迁移文件。
3. 后端基础目录分层。
4. 数据访问规范。

### 6.6 验收标准

1. 可以通过迁移命令生成数据库文件。
2. 数据模型字段与文档一致，没有核心字段缺失。
3. 新增业务功能时不需要重写基础数据库结构。

## 7. 阶段 3：认证系统与前端登录流程

这个阶段的目标是先打通最基础的身份链路，因为世界管理、聊天、VTT 都依赖登录态。

### 7.1 后端实现顺序

1. 实现注册接口。
2. 实现登录接口。
3. 使用 bcrypt 对密码加密。
4. 使用 JWT 生成 token。
5. 实现获取当前用户接口。
6. 实现鉴权中间件。

### 7.2 注册接口细节

至少处理以下逻辑：

1. 用户名不能为空。
2. 用户名唯一。
3. 密码长度最低限制。
4. 密码不明文入库。
5. 注册成功后返回用户基础信息。

### 7.3 登录接口细节

1. 用户名不存在时给出统一错误。
2. 密码错误时给出统一错误。
3. 成功后返回 access token。
4. 返回前端需要的 user 基础资料。

### 7.4 前端实现顺序

1. 注册页面表单。
2. 登录页面表单。
3. 表单校验。
4. 登录成功后保存 token。
5. 全局状态保存当前用户。
6. 路由守卫，未登录跳转登录页。

### 7.5 本阶段输出物

1. 注册/登录接口。
2. 登录态持久化。
3. 前端受保护路由。

### 7.6 验收标准

1. 用户可完成注册和登录。
2. 刷新页面后仍可恢复登录状态。
3. 未登录用户无法访问大厅和世界页。

### 7.7 注意事项

1. 第一版可以用 localStorage，但代码中要把未来迁移 httpOnly Cookie 的边界留出来。
2. 不要把 JWT 解析逻辑分散到各处，统一放到请求层或状态层。

## 8. 阶段 4：大厅与世界管理

这一阶段开始进入业务主流程，目标是让用户能真正创建并进入一个世界。

### 8.1 后端先实现的数据能力

1. 创建世界。
2. 查询世界列表。
3. 查询单个世界详情。
4. 加入世界。
5. 查询当前用户加入的世界。

### 8.2 世界模型第一版建议字段

1. id。
2. name。
3. description。
4. ownerId。
5. visibility。
6. passwordHash，可选。
7. createdAt。
8. updatedAt。

### 8.3 世界创建逻辑

1. 创建 World 记录。
2. 自动创建 WorldMember 记录，把创建者设为 GM。
3. 自动创建一个默认 Scene，避免世界页进入后没有场景。

### 8.4 世界加入逻辑

第一版建议这样做：

1. PUBLIC 世界直接加入。
2. PASSWORD 世界验证密码后加入。
3. FRIENDS 和 PRIVATE 先保留枚举，逻辑可先返回未开放或仅内部使用。

### 8.5 前端页面实现顺序

1. 大厅页显示世界列表。
2. 创建世界弹窗或独立页。
3. 加入世界操作。
4. 我加入的世界列表。
5. 世界详情页基础信息展示。

### 8.6 本阶段输出物

1. 世界创建完整链路。
2. 世界列表展示。
3. 世界加入完整链路。
4. 默认世界页路由。

### 8.7 验收标准

1. 已登录用户能创建世界。
2. 已创建世界会出现在大厅或我的世界列表中。
3. 用户能加入允许访问的世界。
4. 世界创建者进入世界后具备 GM 身份。

## 9. 阶段 5：全局聊天与基础 WebSocket 架构

这是第一阶段的实时能力建设，目标是让整个系统的实时通信基础先稳定下来。

### 9.1 这一阶段不直接做复杂 VTT

先做最简单的文本聊天，因为它最容易验证 Socket 是否稳定、房间机制是否正确、用户映射是否正确。

### 9.2 Socket 基础设计

至少要明确以下概念：

1. 连接时如何携带 token。
2. 服务端如何校验 socket 身份。
3. 如何维护 userId 到 socketId 的映射。
4. 全局房间如何组织。
5. 世界房间如何组织。

### 9.3 本阶段建议先实现的事件

1. connection。
2. disconnect。
3. global:join。
4. global:message:send。
5. global:message:new。

### 9.4 消息持久化范围

第一版建议全局聊天消息可以先持久化到 Message 表，也可以先只保留最近若干条。如果时间紧，先做实时收发，再补持久化。

### 9.5 前端实现

1. 初始化 socket 客户端。
2. 登录后连接 socket。
3. 大厅页右侧或底部显示全局聊天室。
4. 支持发送消息与实时接收消息。
5. 断线重连后恢复连接。

### 9.6 本阶段输出物

1. 可用的 socket 鉴权机制。
2. 全局聊天室。
3. 实时消息广播。

### 9.7 验收标准

1. 两个浏览器窗口之间能实时看到消息。
2. 未登录用户不能建立有效聊天连接。
3. 用户断线后不会导致系统状态混乱。

## 10. 阶段 6：最小 VTT 场景与 Token 实时同步

这是 MVP 的核心阶段。目标不是做完整 Foundry 替代品，而是做一个足够稳定的最小画布闭环。

### 10.1 先定义第一版范围

只做以下内容：

1. 一个场景画布。
2. 一个默认背景或空白背景。
3. 一个可拖拽 Token。
4. Token 拖拽时同步位置。
5. 其他客户端能实时看到变化。

### 10.2 本阶段暂时不做的内容

1. 大量 Fabric 对象同步。
2. 测量尺。
3. 高级绘图工具。
4. 复杂权限控制。
5. 背景图片上传。
6. 角色卡联动。

### 10.3 后端要做的事情

1. 实现 world:join 事件。
2. 实现 scene:state:get 或初始状态接口。
3. 实现 scene:token:move 事件。
4. 校验只有世界成员可以加入该房间。
5. 存储 Token 的基础状态，至少包含坐标与标识。

### 10.4 前端要做的事情

1. 在世界页初始化 Fabric canvas。
2. 加载默认 Token。
3. 用户拖拽 Token 时发出同步事件。
4. 接收到他人事件时更新本地对象。
5. 处理本地拖拽与远程更新冲突。

### 10.5 状态同步原则

1. 只同步 Token 的必要信息，例如 tokenId、x、y、rotation、scale。
2. 不在每次细小操作时广播整个场景 JSON。
3. 保留后续“阶段性保存整张场景 JSON”的能力，但不在第一版就频繁使用。

### 10.6 本阶段输出物

1. 世界页最简 VTT 画布。
2. 一个可拖拽 Token。
3. 跨客户端同步移动。

### 10.7 验收标准

1. 两个用户同时进入同一世界能看到同一个 Token。
2. 任一用户拖动后，另一个用户界面能及时更新。
3. 不同世界之间互不串房。

### 10.8 这是第一版 MVP 的主要验收点

当阶段 3、4、5、6 全部完成后，项目就已经具备第一个真正可演示的最小版本。

## 11. 阶段 7：场景系统、角色卡与世界内聊天

这一阶段的目标是把“可演示的最小版”提升为“可初步使用的跑团版”。

当前执行状态（2026-03-13）：阶段 7 核心链路已完成，已覆盖世界内聊天、多频道、角色卡基础 CRUD、角色与 token 绑定、多 Scene 切换与隔离、场景管理以及关键自动化测试。本阶段后续只保留体验增强项，不再作为主链路阻塞项。

### 11.1 场景系统扩展

1. 支持一个世界多个场景。
2. 支持 GM 切换场景。
3. 支持保存和读取场景数据。

## 12. 阶段 8：AI 酒馆与 GM 助手

这一阶段才开始接 AI，因为这时基础业务结构已经稳定，AI 只需要插入到现有角色卡和世界体系中。

进入条件补充（2026-03-13）：阶段 8 暂不开工。开始实现酒馆或 GM 助手前，必须先做一次需求确认，明确 AI NPC 交互方式、GM 助手输入输出边界、模型接入策略和历史记录范围。

### 12.1 AI 酒馆实现顺序

1. 给 Character 增加 aiConfig 的实际使用逻辑。
2. 在世界内增加酒馆入口页面。
3. 展示启用了 AI 的 NPC 列表。
4. 玩家选择 NPC 后发送消息。
5. 后端拼接 prompt，调用模型 API。
6. 保存 AiSession 对话历史。

### 12.2 GM 助手实现顺序

1. 新增 /api/gm/describe 接口。
2. 接收关键词、场景上下文、世界上下文。
3. 调用模型生成描述。
4. 前端提供输入框与插入按钮。

### 12.3 AI 接入前必须先准备好的东西

1. 环境变量中配置 API Key。
2. 服务端统一的 AI 调用服务层。
3. 请求超时处理。
4. 调用失败的兜底错误提示。
5. 基础限流策略。

### 12.4 本阶段注意事项

1. 不要把 AI 调用代码直接写进 controller。
2. 不要把 prompt 拼接逻辑散落在多个文件。
3. 要明确用户消息、系统设定、NPC 设定、历史消息分别如何组织。

### 12.5 本阶段输出物

1. AI NPC 对话功能。
2. GM 场景描述辅助。
3. AI 对话历史存储。

### 12.6 验收标准

1. 玩家能与指定 NPC 连续对话。
2. GM 能生成场景描述并复制或插入使用。
3. 当模型不可用时，系统不会拖垮主业务流程。

### 12.7 阶段 8 设计补充（2026-03-13）

1. 阶段 8 采用“AAF 主控 + SillyTavern 适配层 + 模型 API”结构，不直接嵌入酒馆复杂前端。
2. 日常操作全部留在 AAF 内，酒馆只负责 AI 编排与模型调用。
3. 总结器改为主持人手动触发，结果先进入待审核草案，不允许自动覆写正式设定。
4. 本阶段详细实施蓝图见：阶段8执行任务清单.md。

## 14. 每次开发时的标准执行顺序

后续每次让 AI 开发具体功能时，建议强制按下面顺序执行：

1. 先说明当前阶段和目标。
2. 先确认要改哪些文件或要新建哪些文件。
3. 先实现后端数据与接口。
4. 再实现前端页面与状态流。
5. 再实现 Socket 或实时同步逻辑。
6. 再跑基础验证。
7. 最后更新 AI 上下文记忆文档。

这样做的原因是：如果顺序反了，常常会先做 UI，最后才发现后端边界不对，导致返工。

## 16. 每阶段完成后必须记录的内容

每次阶段结束后，必须在 AI 上下文记忆文档中追加以下内容：

1. 当前完成到哪个阶段。
2. 新建了哪些目录或核心文件。
3. 新增了哪些接口。
4. 新增了哪些数据库表或字段。
5. 新增了哪些 Socket 事件。
6. 当前能跑通哪些用户操作。
7. 当前已知问题是什么。
8. 下一步应该从哪一个最小任务接着做。

## 17. 当前建议的实际开发起点

如果现在开始正式写代码，推荐直接进入下面这个最小顺序：

1. 创建 monorepo 基础目录。
2. 初始化 client 和 server。
3. 建立 Prisma + SQLite。
4. 实现 auth 模块。
5. 实现 world 模块。
6. 接入全局聊天。
7. 做最小 Fabric VTT 页面。
8. 接入 world 内同步。

不要一开始就做好友系统、BGM、骰子、AI 酒馆、复杂场景编辑器。这些功能应该等到主链路稳定后再加。

## 18. 文档使用方式

后续任何一次新的 AI 对话，如果是要继续开发这个项目，建议同时提供：

1. 技术栈文档。
2. 项目概述文档。
3. 本渐进式开发文档。
4. AI 上下文记忆文档。

其中：

1. 技术栈文档负责告诉 AI 用什么技术。
2. 项目概述负责告诉 AI 为什么做这个项目、核心功能是什么。
3. 渐进式开发文档负责告诉 AI 当前应该先做什么、后做什么。
4. AI 上下文记忆文档负责告诉 AI 已经做到哪一步，不要重复开发。

如果严格按这四份文档协作，后续跨多个聊天持续开发时，效率会明显更高。
