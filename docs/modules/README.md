# AAF 模块化开发导航

本文档用于节省上下文和 token。以后不要默认扫描整个项目；先按任务类型打开对应模块卡片，再按卡片列出的入口文件定位。

## 使用方式

1. 判断用户这次说的是哪个模块。
2. 只打开该模块卡片和它列出的入口文件。
3. 如果跨模块，先读主模块，再读被依赖模块，不要反向全局搜索。
4. 修改完成后，同步更新对应模块卡片的“当前约定”或“待拆分点”。
5. 世界页相关改动还要同步更新 `docs/世界模板实现蓝图.md`。

## 并发 AI 分工规则

1. 每个 AI 只认领一个主模块，并在动手前确认自己的写入范围。
2. 跨模块数据先走 `09` 共享契约；视觉 token 先走 `11` 主题系统；世界壳装配先走 `01` 或 `02`。
3. `WorldPage.tsx` 只做页面装配、状态汇聚和模块入口分发；新业务逻辑优先下沉到对应组件、hook、store、service 或 shared 类型。
4. 新增 tab、资源类型、覆盖层、规则节点、动作类型时，优先扩展集中 registry/config，不要在多个文件里复制字符串和 switch 分支。
5. 并发分支清理只删除“已合并、无 worktree、无未提交改动”的本地分支；脏 worktree 必须保留并交给用户确认。

## 当前优先级分区

- P0 世界内可玩性：`01`、`02`、`03`、`04`、`05`、`06`。
- P1 自动化资源生产：`03`、`04`、`09`。
- P1 生活职业骨架：`07`、`09`。
- P2 大厅与社交：`08`。

## 不要做的事

- 不要为了一个按钮改动扫描 `client/src` 全部文件。
- 不要把系统板、资源编辑器、角色卡、战斗结算继续堆回同一个巨型容器。
- 不要把规则书全文塞进上下文。只在规则冲突时读取相关章节或 `docs/规则规格-战斗与造物系统-v1.md`。
