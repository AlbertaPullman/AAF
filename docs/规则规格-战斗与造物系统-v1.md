# AAF 规则规格 - 战斗与造物系统 v1

更新时间：2026-03-15
状态：讨论稿（可持续增量修订）

## 1. 目标与范围

本文档用于固化当前已确认的规则方向，覆盖：

1. 规则资产分层（后台基础包、扩展包、GM 私设包）。
2. GM 与玩家权限边界。
3. 能力系统与能力联动机制。
4. 战斗面板交互规范。
5. DND 轮次动作经济与资源系统。
6. 范围施法与测量板规则。
7. 移动速度与移动力规则。
8. 造物系统（词条拼装、槽位、宝石、跨时间推进、自动造物）。

## 2. 规则资产分层

### 2.1 平台规则层（后台）

1. 平台维护基础规则资产：种族、职业、物品、能力、天赋、词条模板。
2. 平台可维护扩展包规则资产。
3. 扩展包按包级别启停，不支持玩家自由改写。

### 2.2 世界规则层（GM）

1. GM 可在世界内定义附加规则（私设）。
2. GM 私设可导出为规则包，并可导入其他世界复用。
3. 世界规则结算优先级：基础包 -> 启用扩展包 -> 世界私设包。

### 2.3 玩家使用层

1. 玩家只能使用已启用规则资产。
2. 玩家不可编辑能力定义、词条效果或规则触发器。
3. 玩家可做装配行为（例如把能力加入角色卡、把词条拼装到造物方案）。

## 3. 权限与安全边界

### 3.1 权限矩阵

1. 平台后台：可编辑基础包与扩展包定义。
2. GM：可编辑世界私设、世界能力、世界词条；可导出规则包。
3. 玩家：仅可选择与使用；不可修改规则定义。

### 3.2 服务端裁判原则

1. 所有战斗与造物结算以服务端为准。
2. 前端仅做交互与展示，不具备规则最终解释权。
3. 每次能力施放、造物推进都做服务端二次校验。

## 4. 能力系统模型

### 4.1 能力定义

能力是战斗中可执行的动作单元，最终表现为可点击按钮或被动触发单元。

1. 主动能力：由玩家点击触发。
2. 被动能力：由规则时机自动触发。
3. 能力来源：职业能力、天赋能力、通用能力、装备赋能能力。

### 4.2 能力执行工作流

每次能力执行必须生成 `AbilityWorkflowRun`，并按阶段记录声明、目标确认、资源、反应窗口、攻击/豁免、伤害、应用伤害、应用效果、后处理和完成日志。

自动化模式分三档：

1. `manual`：手动预览，不真实扣资源、扣血或应用效果。
2. `assisted`：半自动，默认内部测试模式，自动处理当前已实现的结算，同时保留审计和撤销快照。
3. `full`：全自动，未来用于确认目标后自动完成整套结算。

触发器和响应能力必须挂在 workflow 阶段内扩展，不允许绕过 workflow 直接改写角色状态。

### 4.3 联动机制（核心）

能力联动通过触发器系统实现，不通过硬编码 if-else 绑定具体能力名称。

一个被动联动能力至少包含：

1. trigger：触发时机。
2. condition：触发条件（标签、来源、状态、资源等）。
3. effect：触发后效果（追加攻击、增伤、改判定、附加状态）。
4. limit：触发次数限制（每回合/每轮/每战斗）。
5. priority：多个联动同时触发时的执行顺序。

### 4.4 联动示例

示例：全力攻击 + 强力攻击。

1. 全力攻击：主动能力，执行多段攻击。
2. 强力攻击：被动能力，在满足条件时追加一次攻击。
3. 引擎流程：主动能力结算 -> 被动触发检查 -> 追加攻击入队。

### 4.5 防循环保护

1. 同一动作链中，同一被动来源仅能触发一次。
2. 动作链触发深度设上限。
3. 触发日志记录 source 与 depth，便于审计。

## 5. 战斗面板规范

### 5.1 面板定位

战斗面板是战斗场景中的可召唤/可隐藏快捷 UI，目标是降低高等级能力检索成本。

### 5.2 页签结构

右侧竖向书签切页，首版最少三页：

1. 能力页：快捷栏按钮 + 关键战斗资源（生命等）。
2. 简易背包页：战斗常用道具快速使用。
3. 状态详情页：增益、减益、异常状态；悬停可见来源与描述。

### 5.3 快捷栏配置

1. 玩家可将角色卡中的主动能力拖拽到快捷栏。
2. 快捷栏只保存布局，不改变能力定义。
3. 被动能力默认不进入按钮区。

## 6. 回合与动作经济

### 6.1 轮次结构

1. 采用先攻顺序进行回合交替。
2. 所有参与者完成一次回合后，轮次 +1。

### 6.2 动作资源

每角色在自己回合内拥有：

1. 1 个标准动作。
2. 1 个快速动作。
3. 1 个机动动作。
4. 无数个自由动作。

组合规则：1 个标准动作 + 1 个机动动作可组合为复合动作。

回合外资源：1 个反应动作。

### 6.3 核心数值资源

1. 生命值：生存资源。
2. 魔力值：法术相关能力消耗，休整恢复。
3. 技力：超魔法或卓越战技消耗。
4. 战意：战系能力强化消耗。

## 7. 多段攻击与中断续攻

### 7.1 全力攻击交互

1. 点击全力攻击后弹出配置面板。
2. 玩家选择本次额外攻击次数。
3. 每 1 次额外攻击消耗固定战意（基础为 5，可被职业特性修正为 4 或 3）。

### 7.2 目标死亡后的续攻

1. 若目标在未打完攻击次数前死亡，显示非阻塞提醒弹窗。
2. 弹窗显示剩余攻击次数。
3. 玩家点击继续攻击后可重新选取目标并结算剩余次数。
4. 多目标与范围攻击需支持按规则的目标重选逻辑。

### 7.3 多次攻击且不同目标的处理

当一次能力包含多次攻击且允许换目标时，采用“攻击计划 + 攻击段队列”模型。

阶段 A：攻击计划（玩家意图层）

1. 发动能力时先生成 N 个攻击段（每段代表一次攻击次数）。
2. 玩家可在结算前一次性分配目标，例如 AABCC（第1/2段打A，第3段打B，第4/5段打C）。
3. 若玩家不做分配，默认全部指向当前主目标。

阶段 B：攻击结算（规则执行层）

1. 每个攻击段按计划顺序结算，独立进行命中、伤害与触发。
2. 每段开始结算前再次校验目标合法性（距离、视线、可达、存活）。
3. 若目标失效（死亡、离场、不可见），该段进入“待选目标”并提示重选。
4. 若目标仍有效，但玩家想临时切换（例如敌方触发护盾术导致命中率骤降），允许手动重定向到新目标。
5. 重定向不消耗额外攻击次数，但需重新满足该段攻击的合法条件。

交互模式：

1. 推荐默认“快速模式”：先分配多目标，再自动逐段结算，仅在目标失效或玩家主动打断时弹窗。
2. 保留“精确模式”：每段结算前都允许改选目标，适合高策略玩家。

规则约束：

1. 追加攻击、重掷、加值等效果默认绑定到“攻击段 ID”，而不是绑定到整组攻击。
2. 若某效果写明“本次能力仅一次”，则在整组攻击内只可触发一次。
3. 若某效果写明“每次命中触发”，则按每个攻击段独立判定。
4. 若某效果写明“声明目标后不可改变”（例如锁定射击），则该效果覆盖本节重定向规则。

示例（远程临时换目标）：

1. 弓箭手计划 3 段攻击，初始分配为 AAB。
2. 第 1 段结算后，A 触发护盾术。
3. 玩家在第 2 段结算前主动重定向为 C，不损失攻击段。
4. 最终按 ACB 结算。

### 7.4 交互细则（快速模式 / 精确模式）

快速模式（默认）：

1. 玩家先完成攻击段目标分配，确认后自动连续结算。
2. 仅在以下情况弹窗或停顿：
	1. 目标失效（死亡、离场、不可见）。
	2. 资源不足导致预设触发失败。
	3. 命中前关键响应窗口（如声明为“必须确认”的能力）。
3. 结算过程中提供“打断”按钮，玩家可在下一段开始前改选目标或修改触发策略。
4. 右下角持续显示“剩余攻击段数 / 当前段目标 / 预计资源消耗”。

精确模式（可选）：

1. 每个攻击段结算前都进入确认节点。
2. 玩家可逐段选择目标、逐段决定响应能力是否触发。
3. 适合关键战斗或高复杂构筑，不建议作为全局默认。

模式切换与继承：

1. 角色可保存默认模式（快速或精确）。
2. 单次能力可临时覆盖默认模式。
3. GM 可在世界规则中配置默认推荐模式，但不强制覆盖玩家角色个人预设。

## 8. 范围能力与测量板

### 8.1 基本流程

1. 点击范围能力后，在鼠标附近生成测量板预览。
2. 测量板形状由能力定义决定（圆形、锥形等）。
3. 鼠标移动确定落点，滚轮调整朝向。
4. 左键确认施放。

### 8.2 持续能力

1. 持续性能力施放后，测量板可留在地图。
2. 默认名称为能力名称。
3. 玩家与 GM 可修改测量板名称和颜色。

### 8.3 半格覆盖规则

规则要求：覆盖方格面积达到一半即算作用区域。

建议判定方式：

1. 用方格与区域形状做相交面积估算。
2. 若相交面积 / 方格面积 >= 0.5，则判定命中。
3. 被纳入方格用测量板同色高亮提示。

## 9. 动画预留

1. 能力可配置动画钩子。
2. 当前阶段动画可留空，不阻塞结算逻辑。
3. 未来可扩展为先播动画再结算，或动画与结算并行。

## 10. 移动速度与移动力

### 10.1 双数值模型

1. 移动速度：角色面板中的客观属性（如 30 尺步行、90 尺飞行）。
2. 移动力：战斗中可被消耗的本回合行动资源。

### 10.2 获得移动力

1. 通常由机动动作的移动行为获得移动力。
2. 若规则未指定类型，弹窗让玩家在可用类型中选择。
3. 若能力明确指定类型与数值（例如 30 尺步行），则不弹窗。

### 10.3 兼容与限制

1. 飞行移动力可兼容地面移动。
2. 本次获得的移动力类型决定本段移动方式。
3. 若后续再次获得移动力，可重新选择并叠加新的可用移动方案。

## 11. 造物系统

### 11.1 生活职业

玩家可通过生活职业进行造物，例如：

1. 锻造。
2. 鞣织。
3. 珠宝。
4. 附魔。
5. 铭文。

### 11.2 词条拼装

1. 魔法物品由词条拼装形成。
2. 词条内容由后台与 GM 维护。
3. 玩家只能选择词条，不能修改词条效果。

### 11.3 双槽位机制

1. 强化槽位：用于附魔/词条强化，存在容量上限。
2. 宝石插槽：用于镶嵌宝石，不占强化槽位。
3. 强词条占用更多强化槽位，形成强度与容量权衡。

### 11.4 跨时间造物

造物采用工期推进：

1. 每个项目有总造价。
2. 每个角色有当日造物产能。
3. 只有在休整时选择制造并推进，才累计当日进度。

工期计算：

所需天数 = ceil(总造价 / 当日产能)

示例：5000 金币总造价，日产 500，需 10 天。

### 11.5 当前造物列表

1. 角色有未完成造物列表。
2. 休整时可选择一个项目推进当日进度。
3. 未选择制造则当日不推进。

### 11.6 自动造物

1. 可勾选一个未完成项目为自动推进目标。
2. 每次休整时默认推进该项目。
3. 玩家可随时取消自动推进。
4. 若材料不足或条件不满足，自动推进应给出明确失败提示。

## 12. 规则引擎实现建议（无代码）

### 12.1 最小引擎职责

1. 规则读取：读取基础包、扩展包、世界私设。
2. 权限裁判：按操作者身份验证可执行性。
3. 触发执行：主动能力与被动联动统一处理。
4. 状态写回：战斗状态、造物进度、日志。

### 12.2 事件建议

1. combat:ability:declare。
2. combat:ability:resolve。
3. combat:passive:trigger。
4. combat:card:new。
5. crafting:project:progress。
6. crafting:project:completed。

## 13. 待确认清单（下一轮讨论）

1. 额外攻击是否允许超过角色当前攻击上限。（无上限，因为角色本身的攻击次数不会特别夸张，且额外攻击本身就有消耗，过度限制会降低玩法空间）。
2. 反应动作在同一轮内是否可被能力重置。（目前框架内为了防止战力膨胀和结算问题，每个角色1轮只有1个反应，但可能会拓宽限制，可以预先留出接口）
3. 范围覆盖的半格判定是否统一使用几何估算，还是按模板离散格预计算。（建议统一使用几何估算，以提高计算效率和准确性）
4. 自动造物是否允许同时挂多个项目（首版建议 1 个）。不允许，造物同一时间只能制造一件魔法物品
5. 宝石拆卸是否有损耗、是否需要额外工序。宝石和加装和拆卸视为造物的一部分，按照造物规则去计算
6. 造物失败机制是否首版启用（首版建议先不启用失败）。需要启用，规则本身就带有失败规则
7. 移动力跨回合是否清零（建议默认清零）。角色在自己回合内获得的移动力在回合结束后清零，而角色在其他人的回合通过反应能力或其他能力获得的移动力，必须立刻进行移动，例如一名角色在敌人的回合用反应获得了30尺步行速度，此时她必须立刻移动至多30尺，无论是否移动了30尺，这些移动力都会在移动结束后清空。我的想法是，回合外的移动通过一个弹窗出现，弹窗上面描述了触发移动的能力以及获得了多少移动速度，随后角色点击“开始移动”按钮，之后就可以拖拽token进行移动，当停止移动后，玩家将失去本次的移动能力并继续后续的结算。

## 14. 高频触发响应策略（交互降噪）

### 14.1 设计目标

1. 避免每次攻击都弹窗问询导致战斗节奏中断。
2. 保留关键决策点的手动控制能力。
3. 支持玩家按角色流派配置默认触发策略。

### 14.2 三档触发策略

每个可响应能力可配置触发档位：

1. 始终询问：每次进入触发窗口都弹窗确认。
2. 智能询问：仅在收益显著或命中边缘时询问，其余按预设自动。
3. 自动触发：满足条件即自动触发，不弹窗。

说明：默认建议使用“智能询问”，既减少打断也避免资源浪费。

### 14.3 多段攻击批量确认

针对单目标多次攻击，采用批量决策窗口：

1. 一次性生成本组攻击的攻击检定结果预览。
2. 在伤害结算前弹出单个汇总弹窗。
3. 玩家可勾选对第几次攻击使用某响应能力。
4. 确认后统一扣资源并进入伤害结算。

该机制优先用于“攻击检定后、伤害前”触发的高频能力。

### 14.4 预设与条件触发

玩家可对单个响应能力保存战斗预设，预设至少支持：

1. 触发时机（命中前/命中后/伤害前等）。
2. 触发条件（未命中才触发、差值在 N 以内触发、仅首击触发等）。
3. 资源阈值（保留至少 X 点资源，不低于阈值才可自动触发）。
4. 每轮次数上限（例如每轮最多 1 次自动触发）。

### 14.5 牧师能力示例

示例能力：攻击检定后、伤害前，消耗 1 次引导神力，使本次攻击 +10。

推荐默认：

1. 档位：智能询问。
2. 条件：当原检定未命中且距离命中差值 <= 10 时触发。
3. 额外限制：每轮最多自动触发 1 次，其余进入批量窗口或手动确认。

### 14.6 结算与审计要求

1. 自动触发必须写入战斗日志（触发来源、消耗、影响到哪次攻击）。
2. 批量确认窗口需展示“本次将消耗的总资源”。
3. 若资源不足，必须降级为未触发并给出明确提示。

## 16. 剧情事件与扮演交互（GM 即兴驱动）

### 16.1 设计目标

1. 支持 GM 在跑团中即时创建事件，不要求预先写完全部分支。
2. 支持事件选项绑定技能检定与 DC，按成功/失败进入不同结果。
3. 支持事件与聊天频道互相绑定，保留叙事上下文。
4. 支持玩家通过“物语点”提出剧情改写请求，由 GM 审核裁决。

### 16.2 事件基础模型

事件由 GM 在世界内创建，最小包含：

1. 标题（例如：卫兵巡检）。
2. 描述（可为空，允许后填）。
3. 作用范围（全体玩家 / 指定玩家）。
4. 选项列表（每个选项可独立配置检定）。
5. 状态（草稿、进行中、已锁定、已结算、已关闭）。

### 16.3 即时编辑流程（空白事件）

1. GM 可先开一个空白事件弹窗并广播给目标玩家。
2. 玩家端实时看到标题/描述/选项的增量更新。
3. GM 可边主持边补充文本、增删选项、调整 DC 和检定类型。
4. 在玩家执行检定前，GM 可自由编辑；检定锁定后进入受限编辑模式（见 16.8）。

### 16.4 选项与检定规则

每个选项可配置：

1. 检定类型（欺瞒、游说、巧手等）。
2. DC 数值。
3. 检定次数规则：
	1. 仅一次（团队共享一次机会，仅一人可执行）。
	2. 每人一次。
	3. 无限制（由 GM 控制关闭）。
4. 成功结果与失败结果：
	1. 预写结果（标准分支）。
	2. 延迟填写（待检定出结果后由 GM 即时输入）。

### 16.5 检定执行与结算时机

1. 玩家选择选项后进入检定流程。
2. 所有重骰、加值、修正效果结算完毕后，才生成最终检定值。
3. 事件系统根据最终检定值与 DC 给出“成功/失败状态”。
4. 若该选项配置为“延迟填写结果”，则由 GM 在看到最终结果后填写叙事后果并发布。

### 16.6 聊天频道绑定（事件上下文可追溯）

1. 玩家在聊天中扮演发言时，可附带本次事件检定标签。
2. 消息可显示摘要标签，例如【技能检定-欺瞒-结果17】。
3. 鼠标悬停标签可查看关联事件（标题、选项、DC、结果）。
4. 事件面板也应能反查相关聊天消息，形成双向索引。

### 16.7 物语点机制（剧情风暴）

玩家可消耗物语点提出“合理剧情改写请求”。

标准流程：

1. 玩家点击事件下方【物语点-剧情风暴】按钮。
2. 在弹窗中输入改写目的与理由。
3. 提交给 GM 审核（通过/驳回/要求补充）。
4. GM 通过后，可手动修改当前事件选项、后果或 NPC 关系以反映改写。

兼容流程（低自动化）：

1. 玩家在聊天中提出改写请求。
2. 玩家手动扣除角色卡物语点。
3. GM 手动调整事件。

说明：首版允许手动流程长期并存，不强制全自动扣点，以减少 bug 风险并保留跑团仪式感。

### 16.8 锁定与防冲突规则

1. 某选项一旦开始检定，该选项进入锁定，不允许再改检定类型与 DC。
2. 事件正文与其他未执行选项可继续编辑。
3. “仅一次检定”选项在首次检定提交后立即关闭。
4. GM 拥有最终裁决权，可强制改判并写入审计日志。

### 16.9 审计与回放

至少记录以下审计项：

1. 事件创建/编辑/关闭记录。
2. 选项检定记录（执行者、最终值、修正来源、成功/失败）。
3. 物语点请求记录（发起人、理由、GM裁决）。
4. 事件与聊天绑定记录（消息ID <-> 事件ID <-> 选项ID）。

### 16.10 事件留档与结算卡片

1. 每个事件在关闭后必须生成“事件结算卡片”，并推送到聊天频道。
2. 结算卡片最少包含：
	1. 事件标题与简述。
	2. 关键经过时间线（选择了哪些选项、谁执行了检定、结果如何）。
	3. 最终后果摘要（世界状态/NPC态度/道具变化等）。
3. 结算卡片需保留事件ID，支持从聊天卡片反查原事件档案。
4. AI助手读取聊天历史时，优先读取结算卡片而非逐条原始聊天，以提高理解效率。
5. 结算卡片默认由系统生成草稿，GM可在发布前做简要修订。
