## 1. 当前目标

AAF 的世界页不是普通后台管理页，而是一个面向跑团实战的 JRPG 风格虚拟桌面。让 GM 和玩家在同一个"电子游戏式舞台"中完成规则查询、角色操作、战斗结算、剧情推进、世界资源管理和自定义扩展。

最高优先级是"可正常使用的世界模板"。世界内部必须先做到：

- 玩家进入世界后知道自己能做什么、该看哪里、如何返回大厅。
- GM 能用可视化方式配置职业、种族、背景、能力、物品、命刻、牌组、随机表、模块和权限。
- 玩家不需要读代码，不需要开发插件，也能使用被 GM 配置好的资源进行游玩。
- UI 要像游戏界面，而不是管理系统表单，强调王道冒险、清晰反馈、舞台感和队伍感。
- 所有自动化能力最终都要走"可配置触发器 + 条件 + 效果 + 持续时间 + 结算日志"的管线。
- 场景、角色、资源、牌组、命刻支持文件夹层级（独立 `Folder` 表）。

## 2. 世界页容器职责

| 区域 | 职责 | 体验要求 |
|---|---|---|

| 顶部战斗序列栏 | 战斗开始后滑出，显示先攻顺序、当前回合 | 默认隐藏；战斗 active 后向下滑出 |
| 中央舞台 | 场景地图、战斗棋盘、剧情视觉舞台 | 保留最大视觉空间 |
| 右侧系统板 | 聊天、战斗、场景、角色、资源包、系统设置入口 | 玩家看到可用入口，GM 看到完整工作区 |
| 底部 HUD | 当前选中目标的快捷栏、资源条、能力/物品槽（GM 跟随选中切换；玩家锁定主角色或部署单位） | 类 BG3/主机 RPG；详见 §11 |
| 底部 HUD 上方 | **场景工具栏抽屉**（1→2 级，平时收起，单击展开）；详见 §12 | 滑出式抽屉 |

## 3. 角色权限与体验分层

| 身份 | 权限范围 | UI 体验 |
|---|---|---|
| GM | 世界所有管理权 | GM 工作区完整；选中 token 时 HUD 跟随切换；可看怪物完整数据；可手动赐予/剥夺玩家情报 |
| 玩家助手 | 协助现场，不拥有模板所有权 | 工具栏中间集（详 §12）：可测距、放临时光源、添加棋子；不可绘墙、不可改场景配置 |
| 玩家 | 操作自己的角色和部署单位、棋子、HUD、聊天、规则查询、公开资源 | HUD 锁定主角色（除非切到部署）；返回大厅入口必有；玩家工具栏含测距/标注/标记敌人焦点 |
| 旁观者 | 只读公开内容 | 输入/创建/执行/投放禁用；显示只读原因 |

## 4. 系统板设计规则

- 顶部标签只放高频入口：聊天、战斗、场景、角色、资源包、系统。
- 战斗页只常驻战斗指挥、先攻、回合推进；能力结算台与剧情事件板通过按钮或快捷栏弹窗打开。
- 角色页只常驻角色花名册条目；详细字段进入弹窗。
- 条目型 UI 全部支持右键菜单（场景卡、角色条目、先攻条目、棋子、命刻、地图空白处、资源条目）。
- GM 工具窗服务跨库拖拽：职业/种族/能力/物品同时打开不互相阻断。
- 公共悬浮说明卡：悬停约 1 秒显示，3 秒固定；卡内词条嵌套展开；位置自动避开既有卡。**当前实现存在 12px 间距导致 hover 断裂的 bug，详见 `docs/插件与后置功能.md` §A**。

## 5. UI 美术方向

亮色 JRPG 王道冒险风（碧蓝幻想式，不是暗黑/科幻）。蓝、白、橘三色基底，橘色仅用于强调/选中/高风险。详见 `.claude/skills/aaf-jrpg-ui/SKILL.md`。

## 6. 自动化能力系统骨架

详见 `docs/规则规格-战斗与造物系统-v1.md`。要点：

- 动作经济：标准/快速/机动/自由/反应/复合/特殊。
- 法术模型：等级序列 + MP + AP + 学派 + 原初/奥秘/神圣类别。
- 能力来源：种族/冒险职业/生活职业/天赋/装备/物品/背景/自定义。
- 触发器：trigger / condition / effect / limit / priority。
- 同名不叠加（按 effectKey）+ 向下取整 + 至少为 1 + 特例优先。
- 防循环：同源单链最多 5 次，最大深度 10 层。
- 反应能力区分**回合内** vs **回合外**（部署单位活动判定依据，详见 §11.3）。

## 9. 场景管理（FVTT 化）

### 9.1 面板布局

```
┌─────────────────────────────────┐
│ 🔍搜索  ↕排序  📁创建文件夹  ✚创建场景 │
├─────────────────────────────────┤
│ 📁 卡拉塔纳                      │
│   📁 城镇                        │
│   📁 大地图                      │
│   📁 危险生物                    │
│     [缩略图] "被逐之妖"斯卡特     │
│     [缩略图] "复生主教"萨尔       │
│ 📁 战斗场景                      │
└─────────────────────────────────┘
```

- 顶部工具行：搜索 + 排序（名称/创建时间/手动）+ 创建文件夹 + 创建场景。
- 树状文件夹（Folder 表）+ 场景卡（缩略 + 名 + 状态徽记）。
- 拖拽：场景拖到任意文件夹 / 文件夹间嵌套。
- 单击 = 选中；双击 = 打开场景配置弹窗（§9.4）；右键 = 菜单（§9.3）。

### 9.2 缩略图生成（前端）

**前端在用户上传背景图后即时生成 256×256 缩略图 + 转 base64 dataUrl** 直接存到 `Scene.thumbnailUrl`（DB Text 字段）。**算力优化要点**：

- Canvas resize：用 `createImageBitmap` + `OffscreenCanvas`（如可用），失败回退到 `<canvas>`。
- 限制原图最大边 4096px，超出先 downsample 再缩。
- 仅在背景图变更时才重新生成，缓存命中跳过。
- 没背景图：渲染场景类型图标（战斗/探索/休整/地下城/城镇/野外）+ 主题色块兜底，不入库。

### 9.3 右键菜单

| 菜单项 | 行为 | 权限 |
|---|---|---|
| 查看场景 | 打开本地舞台预览，**仅影响当前用户**，不影响其他玩家 | All |

| 切换导航栏显示 | "可用性 → 在导航栏中显示"开关的快捷方式（与"对玩家可见"独立） | GM |

## 10. 通用文件夹机制（Folder 表）

### 10.1 适用范围

✅ 场景、角色、资源模板（能力/物品/职业/种族/背景）、牌组、命刻、随机表。
❌ 聊天频道、状态/苦难、玩家个人快捷栏页。

### 10.2 Prisma 模型

```prisma
model Folder {
  id          String       @id @default(cuid())
  worldId     String
  parentId    String?
  type        FolderType
  name        String
  color       String?
  icon        String?
  sortOrder   Int          @default(0)
  collapsed   Boolean      @default(false)  // GM 偏好
  createdAt   DateTime     @default(now())
  updatedAt   DateTime     @updatedAt
  world       World        @relation(fields: [worldId], references: [id])
  parent      Folder?      @relation("FolderTree", fields: [parentId], references: [id])
  children    Folder[]     @relation("FolderTree")
  scenes      Scene[]
  characters  Character[]
  abilities   AbilityDefinition[]
  items       ItemDefinition[]
  professions ProfessionDefinition[]
  races       RaceDefinition[]
  backgrounds BackgroundDefinition[]
  fateClocks  FateClock[]
  decks       DeckDefinition[]
  randomTables RandomTable[]
  @@index([worldId, type])
  @@index([parentId])
}

enum FolderType {
  SCENE CHARACTER ABILITY ITEM PROFESSION RACE BACKGROUND
  FATE_CLOCK DECK RANDOM_TABLE
}
```

### 10.3 迁移策略

- 现有 `folderPath` 字段保留作迁移源；新增 `folderId`。
- 写一次性 migration：解析 `folderPath` → 自动建 Folder 树 → 设 `folderId` → `folderPath` 留作兼容（不删，3 个版本后删）。
- 拖拽校验：目标 Folder 必须 `worldId` 一致 + `type` 匹配。
- "移出文件夹" = `folderId = null`。

### 10.4 可见性

文件夹本身**不**独立设可见性，跟条目走（条目可见 → 文件夹自动展示该条目）。这避免"文件夹隐藏但条目可见"的逻辑矛盾。

## 11. HUD ↔ 角色卡绑定

### 11.1 GM 视角

- 单击 token → HUD 切换显示该角色完整数据（包括隐私字段）。
- **多选 token（框选）→ HUD 显示空**（不混合多角色信息）。
- **未选 token → HUD 显示 GM 默认角色卡**（GM 若有自己的角色），否则空。
- 选怪物 → HUD 显示完整怪物数据。

### 11.2 玩家视角

- HUD **永远锁定**自己的主角色，不随选 token 切换（避免误操作 + 信息串）。
- **唯一例外**：玩家切到自己部署单位时，HUD 显示部署单位数据。

### 11.3 部署单位与部署卡片

**部署单位定义**：玩家拥有的非玩家角色（召唤物、魔宠、伙伴、坐骑等）。GM 控制的友军 NPC **不算**部署，单独走 GM 工作区。

**约束**：

- 每个玩家可同时拥有 **1 个长期部署 + 1 个临时部署**（共 2 个），新部署顶旧（自动解除）。
- 长期部署：跨战斗保留（如永久魔宠、坐骑）。
- 临时部署：战斗中召唤，结算时按法术规则消失。
- **没有独立先攻**：在玩家自己的回合内行动。
- **回合外行动**：依赖能力或反应（与玩家本人共用同一套机制）。反应能力区分**回合内**vs **回合外**两个 flag，回合外反应可在他人回合触发。

**UI 表现**：

- 队伍栏中，玩家主角色卡**下方**挂一张更小的次级卡（部署卡片），同时只显示当前激活的部署单位（长期或临时各一张）。
- **切换交互**：
  - 单击主角色卡 → HUD 切回主角色。
  - 单击部署卡片 → HUD 切到部署单位。
  - HUD 面板上额外提供一个切换按钮（小图标，点击在主角色 ↔ 部署间快速切）。
- 部署单位的 token 在场景中独立可见，被切换时高亮。

**Prisma 调整**：`Character` 表加 `controllerId: String?`（指向控制者角色 ID）+ `deployType: "LONG_TERM" | "SHORT_TERM" | null`（仅部署单位用）。

### 11.4 HUD 配置存储

**主方案：模板共享**——HUD 配置存在角色**模板**上（如怪物模板），同一模板的多实例共用同一份。

**单实例可拆**：在该实例上设置 `hudConfigOverride: HUDConfig | null`，非空时覆盖模板。

**模板 HUD 必须按怪物动作分类落位**：怪物的能力按"攻击 / 法术 / 反应 / 特殊"分组，HUD 不同区域分别承载，玩家/GM 看到的怪物 HUD 一致。

```ts
interface HUDConfig {
  mode: "combat" | "general";
  // 按动作分类（怪物 HUD 必填，玩家 HUD 可选）
  actionGroups: {
    attack: HUDSlot[];
    spell: HUDSlot[];
    reaction: HUDSlot[];
    special: HUDSlot[];
  };
  // 玩家自定义快捷栏（多页）
  hotbarPages: HUDPage[];
  generalTabs: { id, label, type, visible }[];
}
```

## 12. 工具栏分层（HUD 上方滑出抽屉）

### 12.1 形态

## 14. 情报系统

### 14.1 双层存储

- **战斗实例层**（短期）：怪物 token 实例的 `revealedTo: { userId, fields: string[] }[]`。
- **玩家长期记忆**（长期）：`character.intelligenceLog: { monsterTemplateId, fields, gainedAt }[]`。

### 14.2 揭示流程

1. 玩家用情报能力（鉴别等），声明目标怪物。
2. 服务端跑判定 → 命中给出可揭示字段集。
3. 写入 `monsterInstance.revealedTo` 与 `playerCharacter.intelligenceLog`。
4. **同种族共享**：长期记忆按 `monsterTemplateId` 索引；遇到同种族新实例自动应用已知字段。

### 14.3 揭示粒度

**按字段**（不是整张卡）。每个隐私字段独立揭示。

### 14.4 销毁触发

- **怪物死亡瞬间销毁该实例的 `revealedTo`**（战斗内层）。
- 战斗结束**不**额外销毁。
- 同名怪物多实例**独立销毁**（一个死了不影响其他实例的揭示状态）。
- 长期记忆 `intelligenceLog` 不受怪物死亡影响。

### 14.5 GM 手动覆盖

GM 可在角色卡管理弹窗里：

- 手动赐予某玩家对某怪物模板的情报（剧情交代）。
- 手动剥夺玩家长期记忆（失忆 buff、剧情事件）。

## 16. 多职业 + 施法者等级

### 16.1 字段结构

```ts
professions: { professionId: string; level: number }[]
```

### 16.2 施法者等级公式

```
原初 = 灵语者等级 + floor(猎魔人等级 / 2)
奥术 = 魔法师等级 + 吟游诗人等级 + floor((魔能使等级 + 机兵士等级) / 2)
神圣 = 祭司等级 + floor(骑士等级 / 2)
```

向下取整，至少为 1（若有任意全/半施法者职业 ≥ 1 级）。

### 16.3 缓存策略

缓存到角色卡 `casterLevels`。每次 `professions` 数组变更（学新职业 / 升级）触发重算。

## 17. 苦难等级 × 分类

### 17.1 苦难实例结构

```ts
interface SufferingInstance {
  id: string;
  templateId: string;
  name: string;
  tier: 1 | 2 | 3 | 4 | 5;           // 苦难本身的等级
  category: "disease" | "curse" | "poison";
  symptoms: { id: string; name: string; stack: number }[];  // 多症状各自层数
  removalCondition: ConditionExpression;
  appliedAt: string;
}
```

### 17.2 层数语义

- **苦难本身的等级 = `tier`**（1 弱 / 5 强）。
- **症状层数 = `symptoms[].stack`**（如燃烧 3 层，每回合 -1 层 + 造成 N×层数 伤害）。
- 同一苦难内多个症状，每个独立管理 stack。

### 17.3 抗性矩阵应用

判定时取角色 `defense.resistance.suffering`：

- 若 `fullImmune: true` → 全免疫 1-5 级所有分类。
- 否则按 `immune[category] >= tier` → 免疫；`resist[category] >= tier` → 抗性（优势 + 伤害抗性，但不与伤害类型抗性叠加）。
