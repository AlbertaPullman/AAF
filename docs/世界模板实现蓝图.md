# AAF 世界模板实现蓝图

> 本文档是世界页长期开发的上下文锚点。之后每次修改世界模板、权限、自动化结算、UI 布局、生活职业或资源编辑器，都必须同步更新本文档，避免实现方向漂移。

## 1. 当前目标

AAF 的世界页不是普通后台管理页，而是一个面向跑团实战的 JRPG 风格虚拟桌面。核心目标是让 GM 和玩家在同一个“电子游戏式舞台”中完成规则查询、角色操作、战斗结算、剧情推进、世界资源管理和自定义扩展。

最高优先级是“可正常使用的世界模板”。大厅、公开展示页、账号社交页可以后置，但世界内部必须先做到：

- 玩家进入世界后知道自己能做什么、该看哪里、如何返回大厅。
- GM 能用可视化方式配置职业、种族、背景、能力、物品、命刻、牌组、随机表、模块和权限。
- 玩家不需要读代码，不需要开发插件，也能使用被 GM 配置好的资源进行游玩。
- UI 要像游戏界面，而不是管理系统表单，强调王道冒险、清晰反馈、舞台感和队伍感。
- 所有自动化能力最终都要走“可配置触发器 + 条件 + 效果 + 持续时间 + 结算日志”的管线。

## 2. 世界页容器职责

| 区域 | 当前职责 | 体验要求 |
| --- | --- | --- |
| 左上命刻盘 | 显示当前命刻、倒计时或未读总量；GM 可创建/推进/删除命刻 | 要像剧情压力计或 Boss 机制计时器，而不是普通数字控件 |
| 顶部战斗序列栏 | 战斗开始后滑出，显示先攻顺序、当前回合、玩家自己的棋子 | 默认完全不可见；战斗 active 后向下滑出，当前行动者卡片放大并带橙黄色光晕，末尾用细红线标记下一轮循环 |
| 中央舞台 | 场景地图、战斗棋盘、剧情视觉舞台、未来可播放角色立绘/演出 | 保留最大视觉空间，网格和光效不要抢过角色与棋子 |
| 右侧系统板 | 聊天、战斗、场景、角色、资源包、系统设置入口 | 玩家看到可用入口，GM 看到完整控制台；所有按钮要说明它“会做什么” |
| 底部 HUD | 玩家角色快捷栏、资源条、能力/物品槽、日志入口 | 类似 BG3/主机 RPG 的战斗面板，强调可点击、可读、可反馈 |
| 左侧中栏 | 队伍角色简卡，展示 HP、MP、AC、体力、等级、职业、绑定玩家 | 至少舒适显示 4 张卡片，多余滚动；不得再放系统切换按钮 |
| 左下在线列表 | 在线成员、身份、延迟、绑定角色 | 主要用于社交状态和排查掉线，不抢角色简卡职责 |
| 工具窗/浮层 | 资源编辑器、角色管理、权限管理、规则查询、悬浮信息卡 | GM 编辑态要强，玩家查询态要轻；工具窗不使用遮罩和点击外部关闭，允许多开和拖拽摆放 |

## 3. 角色权限与体验分层

| 身份 | 权限范围 | UI 体验 |
| --- | --- | --- |
| GM | 世界所有管理权，含资源模板、模块、权限、场景、命刻、战斗、NPC、隐藏信息 | 系统页显示完整 GM 控制台；编辑按钮明确、集中、有危险操作提示 |
| 玩家助手 | 协助管理现场节奏，可查看更多信息，可辅助测距/绘制/部分现场操作，但不应拥有世界模板所有权 | 更像“副主持/战斗书记”，帮助推进局内事务，避免误改底层模板 |
| 玩家 | 操作自己的角色、棋子、HUD、聊天、规则查询、公开资源、自己的角色资料 | 不看到 GM 管理按钮；必须有返回大厅入口；默认聚焦角色与行动 |
| 旁观者 | 只读观看公开场景、聊天、战斗序列、公开角色和公开规则 | 输入、创建、执行能力、投放棋子等动作禁用，并显示只读原因 |

当前已完成的关键修正：

- `system` 标签对 OBSERVER 及以上可见，避免玩家无法进入系统页返回大厅。
- 系统页中公开“规则查询”和“返回大厅”对所有身份可见。
- GM 控制台只对 GM 展示，玩家不会看到权限管理、资源包导入、模块开关等管理入口。
- 角色面板、棋子面板、能力执行面板已接入身份判断，旁观者只读，玩家只能操作自己的资源。
- GM 可进入本地玩家视角调试模式。该模式只把当前客户端的前端身份视为 PLAYER，用于检查玩家可见入口和禁用态，不改变服务端成员身份，也不影响其他用户观看。

## 4. 系统板设计规则

系统板是世界的“右侧游戏菜单”，不是简单后台。它的布局规则如下：

- 顶部标签只放高频入口：聊天、战斗、场景、角色、资源包、系统。
- 系统板内禁止把低频功能堆成长列表。每个标签只保留当前场景最核心的操作，其余能力结算、剧情事件、角色详情、资源编辑、成员管理都应进入弹窗或专门工作台。
- 战斗页只常驻战斗指挥、先攻序列和回合推进；能力结算台与剧情事件板通过按钮或快捷栏弹窗打开。
- 角色页只常驻角色花名册条目；点击或右键角色条目打开角色卡弹窗，详细字段和保存按钮不直接铺在系统板内。
- 条目型 UI 要支持右键菜单：角色条目、先攻条目、棋子、命刻、地图空白处都应提供就地操作，减少“先选中再移动到固定按钮”的来回折返。
- 玩家系统页只显示规则查询、返回大厅、必要说明。
- GM 系统页显示公开规则查询，然后显示 GM 控制台。
- GM 玩家视角调试时系统页必须显示退出入口，页面顶部也要保留醒目的退出提示。
- 资源查询按钮使用“能力规则、装备物品、种族血脉、职业天赋、背景资料、牌组、随机表”等玩家能理解的命名。
- 编辑器标题可沿用“模板库/编辑器”，但只有 GM 看到。
- 运行状态、模块开关、资源包导入导出属于 GM 工作区。
- 清空提示、危险删除、权限修改必须有明显视觉等级。
- GM 工具窗必须服务跨库拖拽：职业、种族、能力、物品等窗口可以同时打开，不能用灰色遮罩阻断主体舞台或其他工具窗。
- 公共悬浮说明卡用于能力、状态、资源和规则词条：悬停约 1 秒显示，悬停约 3 秒固定；卡片内部词条可继续嵌套展开，并应选择不遮挡既有卡片的位置。

## 5. UI 美术方向

AAF 参考的是 JRPG 王道冒险，气质更接近碧蓝幻想式明亮幻想与冒险旅程，而不是纯后台、纯科幻或纯暗黑。

- 当前世界页应使用蓝、白、橘作为主视觉：亮色冷蓝和白色作为基底，橘色只用于强调、选中、高风险和关键反馈。
- 不应使用大面积深色底。深色会遮挡文字、降低按钮和图标可读性，也会把 AAF 带偏成暗黑/科幻控制台。
- 使用浅蓝渐变、柔和玻璃面板、淡橘高光、舞台网格、卡片高光来建立“王道冒险界面”的底色。后续优化遵循：

- 使用亮色冷蓝、白色、淡橘作为主视觉，不使用默认灰白后台风，也不使用大面积深夜背景。
- 面板要有层级：舞台最大，HUD 次之，系统板靠右固定，辅助信息靠左。
- 按钮像游戏菜单项，有标题、箭头、状态和 hover 反馈，不像表格里的普通按钮。
- 角色简卡要比功能按钮更显眼，因为跑团中角色状态比系统入口更重要。
- 滚动条、禁用态、只读提示都要融入同一套视觉语言，不能出现刺眼默认浏览器白条，也不能因为装饰压低文字对比度。
- 移动端暂不作为最高优先级，但布局不能完全崩坏；未来需提供平板/窄屏世界模式。

### 5.1 主题包扩展点（2026-04-27 落地）

主题以 token 契约 + 主题包 CSS 实现，已落地 2 个包：

- `jrpg-bright`（系统默认）：蓝/白/橘明亮幻想冒险风格
- `dark-arcane`（暗影秘典）：深紫深蓝奥术风格，Phase 4 验证用

**新增主题包步骤**：

1. 在 `client/src/styles/themes/<id>.css` 写 `[data-theme="<id>"] { ... }`，覆盖需要差异化的 `--surface-* / --text-* / --accent-* / --mod-*` token（其余 token 自动 fallback 到 `_contract.css` 默认值）
2. 在 `client/src/styles/index.css` 顶部加 `@import "./themes/<id>.css";`
3. 在 `client/src/lib/theme.ts` 的 `THEME_PACKS` 数组注册（id / label / description / userSelectable）
4. `ThemePackId` 类型补字面量（可选，但有助类型补全）
5. **Vite 缓存陷阱**：新增 @import 后若主题不生效，先 stop dev server，确认无 5174/5175 双进程残留，再 start

**主题切换路径**：

- 个人偏好：`localStorage.setItem('theme-storage', JSON.stringify({state:{userPreference:'<id>'},version:0}))` + reload
- 世界默认：在世界设置中存 `themePack` 字段（GM 可强制锁定）
- 优先级：GM 强制 > 世界默认（非系统包） > 玩家偏好 > 系统默认

**Phase 4 dark-arcane 验证发现**：

- 0 critical chrome leak（2303 条规则全部通过 token 或 `--jrpg-*` 别名链跟随主题）
- 4 处渐变端点 hex（HP/AP/stamina bar、save 按钮）= artistic saturation companion，保留
- 12 处 `var(--jrpg-*)` legacy 调用 = 通过别名链正常跟主题，技术债待清理
- auth 页保留金色卷轴主题（`--auth-*` 专属 token 族不参与世界主题切换，符合 JRPG skill 的硬性禁忌）

## 6. 自动化能力系统骨架

自动化能力系统是世界模板的核心。目标是让 GM 用可视化表单创建特性、法术、战技、装备效果、职业能力、种族能力和生活职业能力。

基础数据应覆盖：

- 能力来源：种族、冒险职业、生活职业、天赋、装备、物品、背景、自定义。
- 激活方式：主动、被动、开关、反应、触发。
- 动作经济：标准、快速、机动、自由、反应、复合、特殊。旧 `move` / `full-round` 只作为历史数据兼容，不再暴露给新配置。
- 法术模型：AAF 法术使用法术等级序列、MP 费用、法术强度 AP、学派与原初/奥秘/神圣类别，不使用旧式法术槽或升阶模型。
- 触发时点：受到攻击、命中、未命中、造成伤害、受到伤害、回合开始/结束、休息、施法、进入区域等。
- 条件：身份、标签、属性、资源、装备、距离、目标类型、攻击类型、伤害类型等。
- 效果：修改数值、添加状态、移除状态、伤害、治疗、临时 HP、AC 修正、优势/劣势、额外骰、资源消耗、持续时间。
- 持续时间：本次攻击、本回合、若干轮、直到短休/长休、永久、特殊。
- 结算日志：每次自动化触发要可追踪，玩家看结果，GM 能看细节。

例子：受到攻击时用反应获得 +10 AC，持续到本次攻击结束。它应被配置成“反应动作 + 受到攻击触发 + 修改自身 AC + 持续到本次攻击结束”，而不是写死在代码里。

## 8. 生活职业与魔法物品制造骨架

生活职业是“流控部分”，但从世界模板角度需要提前建好可扩展骨架。当前先保留规则空位，不硬编码最终职业清单和数值。

核心设定锚点：

- 人物每提升 1 级，可以选择 1 级生活职业。
- 生活职业与冒险职业类似，也有等级、分类、特性和制造范围。
- 分类包含锻造、工程、炼金、附魔等，最终以规则书为准。
- 玩家通过学习不同生活职业互相补足，每个职业承担若干类别魔法物品的制造工作。
- 魔法物品类似“背包英雄”的空间拼装。稀有度越高，可用空间越多，例：1x1、2x2、3x3、6x6 或更高。
- 强化组件占用不同空间，通过安装到魔法物品网格中提供效果。
- 组件效果最终应复用能力系统的 `EffectExpression`，这样装备强化能进入自动结算。

已新增轻量代码骨架：

- `shared/types/life-crafting.ts`

这个文件当前只定义共享类型，不接入数据库和 UI。它包含：

- `LifeProfessionDefinition`：生活职业定义、分类、最大等级、可制造类别、等级特性。
- `LifeProfessionProgress`：角色在某个生活职业上的等级与已选特性。
- `MagicItemChassisDefinition`：魔法物品基底、稀有度、网格尺寸、基础效果。
- `MagicItemComponentDefinition`：强化组件、占用形状、效果、安装条件。
- `MagicItemAssembly`：已安装组件的成品魔法物品。
- `CraftingRecipeDefinition`：配方、需求等级、材料消耗、检定公式。
- `CraftingJob`：制造任务状态、进度和完成信息。

后续规则补完后，要补的内容：

- 稀有度到网格大小的默认表。
- 生活职业完整分类和等级成长表。
- 制造配方的检定、材料、失败/大成功规则。
- 组件旋转、重叠、禁用格、相邻加成、套装联动的校验器。
- 可视化背包式网格编辑器。
- 魔法物品成品对角色属性、能力和状态机的自动化接入。

## 10. Git 与回滚策略

从本节点开始，世界模板开发必须形成可回滚节点。

- 每次完成一批可运行改动后，更新本文档的上下文记录或待办状态。

## 12. 开发提醒

- 后续任务默认按 `docs/modules/README.md` 定位模块，不再默认扫描整个项目。
- 跨模块任务最多先打开主模块和 1 个依赖模块；如果需要继续扩展，再明确记录原因。
- 不要把世界页做成后台系统。玩家进入后首先应该感到“我在冒险现场”，其次才是“我能打开菜单”。
- 不要把 GM 的复杂度暴露给玩家。玩家只需要行动、查看规则、看队友状态、聊天、返回大厅。
- 不要把能力写死。所有职业、特性、装备、状态、生活职业组件都应走可配置数据。
- 不要为了功能堆按钮。每个容器要有明确职责，按钮必须服务于当前场景。
- 任何新系统都要考虑四个身份：GM、玩家助手、玩家、旁观者。

## 13. 2026-04-19 弹窗与工具面板体验修正

本次修正明确区分“世界主舞台”和“可视化配置工具”两类 UI：

- 世界主舞台继续使用蓝、白、橘的明亮 JRPG 视觉，保留游戏感、舞台感和冒险现场感。
- 配置、查询、能力结算、角色详情、资源编辑器等弹窗不再套用强装饰玻璃卡片，而是转向清晰、克制、低视觉压力的工具面板。
- 弹窗内优先使用表单、列表、键值表、分组标题和稳定的矩形按钮，避免大量椭圆胶囊标签。
- 胶囊标签容易因为文本长度变化而形变，也会让信息密度显得混乱；后续只允许用于少数状态徽记，不应作为主要排版结构。
- 能力结算台已改为工具台结构：能力基础信息使用键值表，资源消耗、效果数量、当前施放者使用事实栏，标签使用短矩形标记。
- 资源编辑器和实体管理器在弹窗内使用平面化白底、浅边框、明确列表选中态；GM 的编辑效率优先于装饰效果。
- 后续新增的自定义面板必须先判断自己属于“游戏舞台”还是“工具面板”。舞台可以好看，工具必须先好用。

## 14. 2026-04-20 模板库与可视化编辑器体验修正

本次修正针对 FVTT 式资源编辑体验，但按 AAF 的零代码配置目标重新组织：

- 模板库不再把详情表单常驻铺在系统板中。默认页面只负责找资源、分资源、选资源，信息密度控制在“名称、分类、摘要、关键标签”。
- 资源支持多级分类，例如 `法术/塑能/火焰`、`职业/战士/反应`。分类先使用 `folderPath` 作为轻量路径字段，后续若需要拖拽排序、权限或图标，再升级成独立 Folder 表。
- 双击条目打开编辑器弹窗；单击只负责选中，右键菜单后续继续扩展复制、移动、导出、归档等就地操作。
- 编辑器页签按使用心智拆分：`展示文本` 面向玩家阅读和 GM 撰写描述；`规则结算` 面向自动化管线；`高级数据` 面向 JSON 兼容、调试与未来迁移。
- 检定方式必须驱动后续字段显隐。以后所有类似字段也遵循这个规则：先选大类，再出现该大类真正需要的配置，不允许把所有可能字段一起铺出来。
- 工具面板继续保持浅色、矩形、低装饰、强对齐。它服务 GM 的长时间录入，不追求舞台 UI 的强幻想装饰。
- 旧实验组件如果暂时不用，必须放到 dormant 分类或不挂载到世界页，避免隐藏 UI 继续消耗运行成本。
